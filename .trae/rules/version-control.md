# 项目开发规范

## 代码风格
- 缩进：2空格，不使用Tab。
- 命名：变量/函数使用 camelCase，常量使用 UPPER_SNAKE_CASE，类使用 PascalCase。
- 注释：关键逻辑必须注释，函数应有 docstring。

## 提交规范
- 格式：`<type>(<scope>): <subject>`，如 `feat(auth): add login api`
- type 可选：feat, fix, docs, style, refactor, test, chore
- 提交应原子化，关联 issue（如 `#123`）

## 文档要求
- 修改功能必须同步更新 README 或相关文档。
- 新增模块需在 README 中简要说明。

## 版本号管理
采用语义化版本 2.0.0，格式：`主版本号.次版本号.修订号`

- **主版本号**：不兼容的 API 修改
- **次版本号**：向下兼容的新功能
- **修订号**：向下兼容的问题修复

**版本号更新规则**：每次对项目进行任何修改后，必须根据修改的重要性更新 `package.json` 中的版本号：
- **主版本号递增**：重大架构变更、不兼容的 API 修改、核心功能重写
- **次版本号递增**：新增功能、新增组件、性能优化、UI 改进
- **修订号递增**：Bug 修复、小改进、代码重构、文档更新

每次发布前，根据更改类型手动或自动递增对应版本号。在构建过程中，需将当前版本号自动更新到 `manifest.json` 的 `version` 字段，确保构建产物中的版本号与项目版本保持一致。

## 测试要求
- 新功能需包含单元测试。
- 修复 bug 应添加测试防止回归。
- 确保现有测试通过，覆盖率不下降。

## 其他
- 优先复用现有代码，避免引入重复依赖。
- 遵循项目已有架构和设计模式。
